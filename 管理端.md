# 管理端 PRD：运营控制台 V2.0（单人可交付版）

> 版本说明：本文档为独立开发者可在 2-3 周内完成的极简管理后台。
> 不做华丽的可视化大屏，只做运营一个人每天必须用到的功能。

---

## 一、产品定位

**访问方式：** 独立路由（如 `/admin`），部署在同一域名下，需要管理员账号密码登录。
**使用者：** 产品运营者本人（MVP 阶段只有一人使用管理后台）。
**核心诉求：** 看收入、处理提现、管理用户、控制基础参数。不需要花哨，需要高效。

---

## 二、管理员登录

**路由：** `/admin/login`

**认证方式：** 固定的管理员账号 + 密码（存在后端环境变量中，不走 Supabase 用户体系）。

**实现：** 登录成功后颁发一个独立的管理员 JWT（签发时添加 `role: admin` claim），有效期 8 小时。所有管理端接口验证此 Token。

**UI：** 极简。白底或深色底，一个输入框组合，不需要和用户端保持一致的视觉风格。管理后台以功能为主。

---

## 三、页面结构

```
管理后台
├── 数据概览（首页/Dashboard）
├── 订单管理
├── 提现管理（最高优先级，影响用户体验）
├── 用户管理
└── 系统设置
```

左侧固定导航栏 + 右侧内容区。桌面端布局，不需要移动端适配（管理员用 PC 操作）。

## 四、数据概览（Dashboard）

**路由：** `/admin/dashboard`

**这一页回答三个问题：今天赚了多少、花了多少、有多少人用。**

### 4.1 核心指标卡片（顶部4个）

| 指标 | 说明 | 数据来源 |
|------|------|----------|
| 今日收入 | 今天所有支付成功的订单金额合计 | orders 表，当日 paid_at |
| 今日订单数 | 今天支付成功的订单数量 | orders 表 |
| 今日新增用户 | 今天注册的新用户数 | users 表 |
| 待处理提现 | 状态为 pending 的提现申请数量 | withdrawals 表 |

**待处理提现数量如果 > 0，显示红色角标提醒。**（这是影响用户体验的最敏感指标）

### 4.2 最近7天收入折线图

X轴：日期（最近7天）
Y轴：当日总收入（元）

使用 Recharts 或 Chart.js 实现，数据从数据库聚合查询。

### 4.3 API 成本估算（简易版）

**说明：** 因为 Dify 会记录每次调用的 token 数，可以在这里做粗略估算。

显示：
- 今日 AI 调用次数（= 今日成功生成报告的订单数）
- 今日 AI 估算成本（调用次数 × 0.8 元，这个系数在系统设置里配置）
- 今日毛利（今日收入 - 佣金支出 - AI成本）

**注意：** 这是估算值，不是精确成本。精确成本要去 Dify / 模型平台后台查。

### 4.4 最近5笔订单（快速查看）

表格展示最新5条订单：用户手机号（脱敏） + 类目 + 金额 + 时间 + 状态。

## 五、订单管理

**路由：** `/admin/orders`

### 5.1 订单列表

**表格字段：**

| 字段 | 说明 |
|------|------|
| 订单号 | order id 前8位（可点击查看详情） |
| 用户 | 手机号脱敏（138****8888） |
| 类目 | 职场困局 / 情感困局 |
| 金额 | ¥28.8 |
| 状态 | 待支付 / 已支付 / 生成中 / 已完成 / 生成失败 / 已退款 |
| 支付时间 | 格式：MM-DD HH:mm |
| 操作 | 查看详情 / 退款（仅已完成状态可退款） |

**筛选项：**
- 日期范围（默认今天）
- 订单状态
- 搜索手机号

**分页：** 每页 20 条，后端分页。

### 5.2 订单详情（弹窗或抽屉）

点击订单行展开详情：
- 用户输入的原始困境（完整文本）
- AI 生成的报告（格式化展示）
- 支付流水号（微信/支付宝返回的）
- 是否有邀请来源（展示邀请人手机号脱敏）

### 5.3 退款操作

点击「退款」按钮：
1. 弹窗确认：「确认对订单 XXXX 执行退款 ¥28.8 吗？」
2. 确认后：
   - 调用微信/支付宝退款 API（原路退回）
   - 订单状态更新为 `refunded`
   - 若该订单曾触发佣金，自动从邀请人可提现余额中扣除对应佣金（¥8.64）
   - 若邀请人已提现，余额不足，允许余额变为负数（下次收益先抵扣）

**退款入口：** 用户端**不开放**退款入口。所有退款由用户联系客服，客服在管理后台手动处理。

**退款操作（管理端执行）：**
1. 确认退款金额和原因
2. 在微信/支付宝商家后台手动原路退款
3. 在管理后台将订单状态改为 `refunded`
4. 若该订单产生过佣金（自购返佣或推荐佣金），手动在备注中记录，由运营人工判断是否从 `wallets` 扣除

**注意：** 不存在用户余额为负数的情况。佣金扣除属于人工判断场景，在 MVP 阶段不做自动化，运营个案处理即可。

## 六、提现管理

**路由：** `/admin/withdrawals`

**这是最重要的管理模块。用户申请提现后如果长时间没有处理，会直接影响口碑和裂变意愿。**

### 6.1 提现申请列表

**表格字段：**

| 字段 | 说明 |
|------|------|
| 申请时间 | MM-DD HH:mm |
| 申请人 | 手机号脱敏 |
| 申请金额（税前） | 用户申请的原始金额 |
| 预计打款金额（税后） | 税后实际应打款金额（见税务说明） |
| 收款方式 | 微信 / 支付宝 |
| 收款账号 | 明文展示（用于打款） |
| 收款人真实姓名 | 明文展示 |
| 状态 | 待审核 / 已通过 / 已拒绝 / 已打款 |
| 操作 | 通过 / 拒绝 |

**默认只展示「待审核」状态，通过顶部 Tab 切换查看其他状态。**

### 6.2 税务处理说明（MVP 简化方案）

MVP 阶段不做自动代扣税，采用以下简化方案：

- 佣金金额 ≤ ¥800/月：按劳务报酬免税处理，打款全额
- 佣金金额 > ¥800/月：管理员手动计算税额，在备注中记录，从打款金额中扣除

后期用户量大后，再对接财税平台（如易代扣）进行自动化处理。

**打款方式（MVP 全部手动）：** 管理员登录微信商户后台或支付宝商家后台，手动发起企业付款到零钱/银行卡。打款后，在系统中点击「确认已打款」，状态变为「已打款」，系统自动扣除用户的可提现余额。

### 6.3 审核操作

**通过：**
1. 点击「通过」按钮
2. 可选填写备注
3. 状态变为「已通过」，等待手动打款

**拒绝：**
1. 点击「拒绝」按钮
2. 必填拒绝原因（下拉选项 + 自定义输入）：
   - 收款账号信息有误，请重新提交
   - 账号存在异常，暂时冻结
   - 其他（手动填写）
3. 拒绝后，用户的可提现余额原路返还（恢复为可提现状态）
4. 用户在 App 收到通知（系统消息）

**确认打款：**
1. 管理员在微信/支付宝后台完成打款后，回到系统点击「确认已打款」
2. 系统扣除用户对应的余额
3. 状态变为「已打款」

## 七、用户管理

**路由：** `/admin/users`

### 7.1 用户列表

**表格字段：**

| 字段 | 说明 |
|------|------|
| 注册时间 | MM-DD HH:mm |
| 手机号 | 脱敏展示 |
| 累计消费 | 该用户所有付费订单金额总和 |
| 邀请人数 | 通过该用户邀请码注册并付费的人数 |
| 累计佣金 | 该用户历史获得佣金总额 |
| 状态 | 正常 / 冻结 |
| 操作 | 查看详情 / 冻结 / 解封 |

**筛选项：** 注册日期范围 / 账号状态 / 搜索手机号

**分页：** 每页 20 条。

### 7.2 用户详情（弹窗）

- 基本信息：手机号、注册时间、微信 openid（是否绑定）
- 消费记录：该用户所有订单列表
- 佣金明细：每笔佣金的来源订单和状态
- 提现记录

### 7.3 账号冻结

**冻结效果：**
- 被冻结用户无法登录
- 被冻结用户的待结算佣金暂停结算
- 被冻结用户的提现申请自动挂起

**使用场景：** 发现刷单、薅羊毛行为（如短时间内大量退款+佣金套现）时使用。

## 八、系统设置

**路由：** `/admin/settings`

**只放真正需要动态调整的配置，不做过度设计。**

### 8.1 价格配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 单次咨询价格 | ¥28.8 | 修改后立即对新订单生效 |
| 分销佣金比例 | 30% | 修改后仅对新产生的佣金生效，历史不影响 |
| 最低提现金额 | ¥50 | 用户需达到此金额才能申请提现 |

### 8.2 AI 成本估算系数

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 单次 AI 调用估算成本 | ¥0.8 | 用于 Dashboard 的毛利估算 |

这个数字需要定期根据实际模型费用校准。

### 8.3 公告设置（可选）

**一个简单的文本框：** 在用户端首页顶部展示公告横幅（如：「平台维护通知」）。留空则不展示。

## 九、核心 API 接口（管理端专用）

所有接口需要 Admin JWT 鉴权。

### 数据概览
- `GET /admin/stats/overview` — Dashboard 核心指标
- `GET /admin/stats/revenue-chart` — 最近7天收入折线图数据

### 订单
- `GET /admin/orders` — 订单列表（支持分页、筛选）
- `GET /admin/orders/{id}` — 订单详情
- `POST /admin/orders/{id}/refund` — 执行退款

### 提现
- `GET /admin/withdrawals` — 提现列表（支持状态筛选）
- `POST /admin/withdrawals/{id}/approve` — 审核通过
- `POST /admin/withdrawals/{id}/reject` — 审核拒绝（需传拒绝原因）
- `POST /admin/withdrawals/{id}/confirm-paid` — 确认已手动打款

### 用户
- `GET /admin/users` — 用户列表（支持分页、筛选）
- `GET /admin/users/{id}` — 用户详情
- `POST /admin/users/{id}/freeze` — 冻结账号
- `POST /admin/users/{id}/unfreeze` — 解封账号

### 设置
- `GET /admin/settings` — 获取所有配置项
- `PUT /admin/settings` — 更新配置项（批量）

## 十、管理端 UI 规范（极简）

管理端不需要沉浸式设计，**以信息密度和操作效率为第一优先级**。

**推荐使用 Ant Design 或 shadcn/ui 组件库**，不做从零自定义设计。

- 配色：浅色模式（白底灰字黑字），顶部导航栏用深色（`#1a1a2e`）
- 字体：系统默认字体
- 表格：标准数据表格，支持列排序
- 弹窗：使用组件库自带的 Modal/Drawer
- 表单：标准表单组件，不做特殊设计

**开发时间估算：** 管理端使用成熟组件库，整体开发时间约 1-2 周。

## 十一、MVP 不做的功能

以下功能不在 V2.0 范围内：

- 自动化晨报（邮件/企业微信推送）
- 复杂的数据下钻（点击数字查看明细）
- 多管理员账号体系
- 操作日志审计
- 动态分润规则引擎（阶梯佣金/限时活动）
- 银企直联自动打款
- 自动代扣个税
- 数据导出（CSV/XLSX）
- AI 内容质量评分
- 用户画像分析
