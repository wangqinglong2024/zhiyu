# 编号生成规则

## 1. 类目编码 `china_categories.code`

```
格式：固定 2 位数字  01..12
来源：迁移文件硬编码（见 09-种子数据.md）
并发安全：无（数据字典，运行时不再增删）
```

---

## 2. 文章编码 `china_articles.code`

```
格式：[A-Z0-9]{12}
长度：12 位
字符集：大写字母 + 数字（去除易混淆字符 I, O, 0, 1 → 实际集合为 ABCDEFGHJKLMNPQRSTUVWXYZ23456789，共 32 字符）
序号重置：全局递增，不重置
并发安全：使用 Postgres RPC 在事务内重试（最多 5 次）直至 INSERT 成功
对外可见：管理后台详情可见；前端 C 端不展示
```

### 实现：`fn_gen_article_code()`

```sql
create or replace function zhiyu.fn_gen_article_code()
returns text language plpgsql as $$
declare
  alphabet constant text := 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'; -- 32 chars
  v_code text;
  v_attempts int := 0;
begin
  loop
    v_code := '';
    for i in 1..12 loop
      v_code := v_code || substr(alphabet, 1 + floor(random() * 32)::int, 1);
    end loop;

    -- 唯一性检查
    perform 1 from zhiyu.china_articles where code = v_code;
    if not found then
      return v_code;
    end if;

    v_attempts := v_attempts + 1;
    if v_attempts >= 5 then
      raise exception 'CHINA_ARTICLE_CODE_GEN_FAILED';
    end if;
  end loop;
end;
$$;
```

> 说明：12 位 × 32 字符空间约 1.15×10^18，碰撞概率极低；保留重试是防御性。

## 3. 句子编码 `china_sentences.seq_no`

```
存储：int（1..9999）
展示：前端 padStart(4,'0') → "0001".."9999"
作用域：每篇文章独立编号（复合唯一索引 (article_id, seq_no)）
生成：每篇文章内顺序递增；批量插入时由 RPC 一次性分配
并发安全：通过 fn_next_sentence_seq() 在表锁中分配
重排策略（PM 答 Q8）：删除/插入/调序后必须保证 seq_no 连续无空洞
```

### 实现：`fn_next_sentence_seq(p_article_id uuid)`

```sql
create or replace function zhiyu.fn_next_sentence_seq(p_article_id uuid)
returns int language plpgsql as $$
declare
  v_next int;
begin
  -- 锁住该文章的所有句子，避免并发分配相同 seq_no
  perform 1 from zhiyu.china_articles where id = p_article_id for update;

  select coalesce(max(seq_no), 0) + 1
    into v_next
    from zhiyu.china_sentences
   where article_id = p_article_id;

  if v_next > 9999 then
    raise exception 'CHINA_SENTENCE_SEQ_OVERFLOW';
  end if;

  return v_next;
end;
$$;
```

### 实现：`fn_resequence_sentences(p_article_id uuid)`（重排）

```sql
create or replace function zhiyu.fn_resequence_sentences(p_article_id uuid)
returns void language plpgsql as $$
declare
  v_row record;
  v_new int := 0;
begin
  -- 锁文章，串行化重排
  perform 1 from zhiyu.china_articles where id = p_article_id for update;

  -- 临时把 seq_no 加上一个大偏移，绕开复合唯一约束
  update zhiyu.china_sentences
     set seq_no = seq_no + 100000
   where article_id = p_article_id and deleted_at is null;

  for v_row in
    select id, seq_no, audio_url_zh
      from zhiyu.china_sentences
     where article_id = p_article_id and deleted_at is null
     order by seq_no asc
  loop
    v_new := v_new + 1;
    update zhiyu.china_sentences
       set seq_no = v_new,
           -- seq_no 变化时，旧的音频缓存键失效，必须重置（PM 答 Q8）
           audio_url_zh       = null,
           audio_status       = 'pending',
           audio_provider     = null,
           audio_voice        = null,
           audio_duration_ms  = null,
           audio_generated_at = null,
           audio_error        = null
     where id = v_row.id;
  end loop;
end;
$$;
```

- 删除句子：`fn_delete_sentence(sentence_id)` 完成软删后调用 `fn_resequence_sentences(article_id)`。
- 中间插入：`fn_insert_sentence_at(article_id, after_seq_no, payload)` 插入到末尾（取 `next_seq`），随后调用 `fn_resequence_sentences` 把目标位置整理到位（或直接以 `after_seq_no + 0.5` 临时排序后再重排）。
- 调序：`fn_reorder_sentences(article_id, ordered_ids[])` 接受新顺序的 id 列表，按列表顺序重写 seq_no。

### 音频缓存键约定

- Storage 路径：`china-tts/<article_code>/<seq_no_padded>.mp3`，例如 `china-tts/A7K2P9X3M4QR/0003.mp3`。
- **缓存键由 `(article_code, seq_no)` 决定，不是句子 UUID**——这样：
  - 同一句子在不同文章间复制时，按新 `(article_code, seq_no)` 重新生成；
  - 重排导致 `seq_no` 变化时，旧 mp3 文件不删（其他用户的浏览器缓存可能仍引用），但 DB 中 `audio_url_zh` 清空、`audio_status` 重置为 `pending`，下次播放重新生成新键的 mp3；
  - 旧 mp3 由 cron `cron_china_purge_orphan_audio`（每周）扫描 Storage 与 DB 比对清理。
- 批量插入 N 条：调用 `fn_bulk_insert_sentences(article_id, payload jsonb[])`，函数内 `for update` 一次锁文章，连续分配 seq_no 后批量 insert，必要时再 `fn_resequence_sentences` 兜底。
