## A15 · 全局搜索（聚合：文章 + 句子）

**对应需求中的操作**：F2-用户 §三 · 管理端「类目层级列表时支持全局所有」

**方法**：GET
**路径**：`/admin/v1/china/search`

**业务校验**：
- `q` 必填，长度 1–60；为空或全空白 → 400 `CHINA_SEARCH_QUERY_TOO_SHORT`。
- `q` 超过 60 字符 → 400 `CHINA_SEARCH_QUERY_TOO_LONG`。

**请求参数**：

| 参数 | 位置 | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| `q` | query | string | ✅ | 搜索关键字（**5 语言全字段**：文章 `code/title_pinyin/title_i18n.{zh,en,vi,th,id}` + 句子 `pinyin/content_{zh,en,vi,th,id}`） |
| `scope` | query | string | ❌ | `all`（默认）/ `articles` / `sentences` |
| `category_code` | query | string | ❌ | 限定类目 `01..12`；缺省 = 全部 |
| `status` | query | string | ❌ | 文章状态：`draft` / `published` / `all`（默认） |
| `page` | query | int | ❌ | 默认 1 |
| `page_size` | query | int | ❌ | 默认 20，上限 50 |

**排序规则**（PM 答 F3-Q2「排序你看着来」）：
1. 命中文章标题/编码 优先于 命中句子内容；
2. 同组内按 `relevance` 降序：精确匹配 > 整词匹配 > 子串匹配；
3. 平局按 `updated_at desc`。

**实现要点**：
- 后端用 PostgreSQL `ILIKE` + `jsonb_path_ops`/`gin_trgm_ops` 索引；句子 `content_{zh,en,vi,th,id}` 已按列拆分（F1 §3），可直接 GIN trigram 索引。
- `highlights[*].snippet`：服务端对命中字段做"匹配片段抽取 + `<em>` 包裹"；前端**只渲染 `<em>`**（白名单），其它 HTML 全部 escape，避免 XSS。
- 受软删过滤：`articles.deleted_at IS NULL` AND `sentences.deleted_at IS NULL`。
- 查询超时 3s，超时返 504 + `CHINA_SEARCH_UPSTREAM_TIMEOUT`（沿用 50400）。

**错误场景**：

| HTTP | code | 触发条件 | 错误信息 |
|------|------|---------|---------|
| 400 | `CHINA_SEARCH_QUERY_TOO_SHORT` (45029) | `q` 为空/全空白 | 请输入搜索关键字 |
| 400 | `CHINA_SEARCH_QUERY_TOO_LONG` (45030) | `q` > 60 | 关键字过长，最多 60 字符 |
| 400 | 40002 | `sort` 越界 | 排序字段不在白名单 |
| 504 | 50400 | 查询超时 | 搜索超时，请缩小关键字范围或稍后重试 |

## 与 C 端的关系

- C 端**不提供**全局搜索接口；只能从「类目 → 搜索本类目」入口（C2 的 `q` 参数）。
- 全局搜索专属于管理端，避免对应用端 RLS 评估造成额外压力。
