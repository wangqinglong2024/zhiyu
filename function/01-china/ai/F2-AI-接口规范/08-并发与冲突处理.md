# 并发与冲突处理

> 来源：F2-用户 §四「如果未来管理员账号增加，多人同时编辑一个文章，以最后保存为准。」

---

## 场景 1：多管理员同时编辑同一文章基本信息（A5）

**策略**：**最后保存为准（Last Writer Wins）** + **前端轻量提示**。

**实现**：
- 后端不加版本号、不返回 409。`PATCH /admin/v1/china/articles/:id` 始终成功，按到达顺序覆盖。
- `china_articles.updated_at` + `updated_by` 由触发器写入，前端在打开编辑页时记录 `loaded_updated_at`；保存成功后用响应里的新 `updated_at` 比较：
  - 若 `response.updated_at > loaded_updated_at + epsilon` 且 `updated_by != self`，前端 toast：「该文章在你编辑期间被 admin01 修改过，已覆盖」。
- 不锁文章；编辑期间允许其他人继续编辑，符合 PM 答复。

---

## 场景 2：多管理员同时编辑同一句子（A12）

同场景 1，最后写入为准；当 `content_zh` 被任一方修改时，音频缓存清空（无并发问题）。

---

## 场景 3：句子重排 / 删除 / 插入并发

**策略**：**串行化**——所有 `fn_resequence_sentences` / `fn_insert_sentence_at` / `fn_reorder_sentences` 在事务内对所属 `china_articles` 行加 `FOR UPDATE`。

- 第二个并发请求等待第一个事务提交后再执行；不会丢失。
- 前端体验：第二个请求耗时略长，但结果正确。

**异常**：极端长事务下可能 statement_timeout（建议 30s），返回 504，前端提示稍后重试。

---

## 场景 4：用户与管理员同时操作（C4 触发 TTS 同时 A12 改 zh）

- A12 修改 `content_zh` 时清空音频缓存（事务内 `UPDATE ... SET audio_url_zh=null, audio_status='pending'`）。
- C4 处于 `processing` 阶段时，A12 的 UPDATE 会等待 C4 完成（行锁）；C4 写入 `ready` 后 A12 立即覆盖回 `pending`，旧音频文件残留由 cron 清理。
- 用户播放时若拿到的是旧 mp3 也无害（已下载片段播完即丢）。

---

## 场景 5：文章 A6 发布与 A8 删除并发

- 二者都对 `china_articles` 同行加锁；后到的发现状态不符合前置条件，返回 409 `CHINA_ARTICLE_STATUS_CONFLICT`。

---

## 场景 6：用户进度 PUT 并发

- 同用户多端并发上报：覆盖即可（最大值不强求）；如需"只允许递增"，应在 `learning_progress` 域内单独处理，不属于本功能范围。

---

## 限流与防重复

| 场景 | 防护 |
|------|------|
| C4 同 IP 频繁触发 TTS | 6 次/min 限流 + 行锁 + `Idempotency-Key` 头 |
| 管理端批量操作（A14） | service-role 单管理员场景，不额外限流 |
| 文章/句子写接口 | 标准 120 r/min（按用户） |
