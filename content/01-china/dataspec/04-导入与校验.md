# 发现中国 · 04 · 导入与校验

> 导入对象为单篇 `*.article.json`。脚本使用 `DATABASE_URL` 连接数据库，可通过 `--env-file` 注入不同环境变量。

---

## 1. 导入脚本

脚本路径：`system/scripts/db/import-china-article-json.mjs`

### 1.1 Dry run

```bash
node system/scripts/db/import-china-article-json.mjs \
  --data-root content/01-china/data/articles/01-history/phase1 \
  --env-file system/docker/env/.env.dev \
  --dry-run
```

### 1.2 正式追加导入

```bash
node system/scripts/db/import-china-article-json.mjs \
  --data-root content/01-china/data/articles/01-history/phase1 \
  --env-file system/docker/env/.env.dev \
  --publish
```

### 1.3 清空发现中国并替换

```bash
node system/scripts/db/import-china-article-json.mjs \
  --data-root content/01-china/data/articles/01-history/phase1 \
  --env-file system/docker/env/.env.dev \
  --replace \
  --publish
```

`--replace` 会物理删除 `zhiyu.china_articles`，并通过外键级联删除 `zhiyu.china_sentences`；同时清理 `zhiyu.learning_progress where source='china'`。仅在本地或明确授权环境使用。

### 1.4 按类目彻底清理并重建

如果要彻底删除指定类目，包括文章、句子和类目记录，再按当前 JSON 重新创建导入类目，使用 `--clear-category-codes`：

```bash
node system/scripts/db/import-china-article-json.mjs \
  --data-root content/01-china/data/articles/01-history/phase1 \
  --env-file system/docker/env/.env.dev \
  --clear-category-codes 01,02 \
  --publish
```

上例会删除中国历史和中国美食两类的文章、句子和类目记录，然后只根据当前历史 JSON 重新创建 `01 中国历史` 并导入历史文章。用于本地重建错误数据，生产环境需谨慎。

如果要一次替换当前已生成的所有发现中国类目，可把 `--data-root` 指向根目录：

```bash
node system/scripts/db/import-china-article-json.mjs \
  --data-root content/01-china/data/articles \
  --env-file system/docker/env/.env.dev \
  --replace \
  --publish
```

## 2. 脚本行为

1. 读取 `--env-file` 与当前环境变量，取得 `DATABASE_URL`。
2. 扫描 `--data-root` 下所有 `*.article.json`。
3. 校验 schema、标题、`source_story_zh`、5 语言、SEO/GEO、句子数量、句子连续性、长度限制。
4. `--dry-run` 仅输出校验统计，不写库。
5. `--replace` 先清空发现中国文章与句子。
6. 每篇文章调用 `zhiyu.fn_gen_article_code()` 生成 code。
7. 插入 `china_articles` 与 `china_sentences`。
8. `--publish` 时调用 `zhiyu.fn_publish_article(article_id)`。

本机没有 `psql` 时，脚本会自动使用 Docker 临时运行 `postgres:16-alpine` 客户端，并加入 `supabase_default` 网络，适配本仓库自托管 Supabase 开发环境。

## 3. 校验规则

| 项 | 规则 |
|----|------|
| 文件 | `schema` 必须是 `china.article.v2` |
| 单篇 | 每个文件只含一个 `article` |
| 类目 | 顶层、article、manifest 的 `category_code` 必须一致 |
| 标题 | `title_pinyin` 1–200，`title_i18n.*` 1–40 |
| 句子 | `sentences.length` 1–120，首期应为 120 |
| 顺序 | `seq_in_article` 从 1 连续递增 |
| 内容 | `pinyin` 1–600，`content_*` 1–400 |
| 多语言 | `zh/en/vi/th/id` 全齐 |
| SEO/GEO | `seo.primary_keywords`、`geo.bluf`、`geo.entities` 必填 |
| 故事源文 | `source_story_zh[n]` 必须等于 `sentences[n].content_zh` |
| 内容安全 | 生成侧必须声明并遵守 `content_policy.risk_controls`，避免东南亚市场敏感议题和成人性内容；正文禁止生成过程话术和 SEO/GEO/AI 元话术 |
| AI 直出 | 全部 `content_zh / pinyin / content_en/vi/th/id / seo / geo` 必须由 AI 在上下文中逐句直出，禁止脚本/字典/模板批量生成；详见 [02-字段与JSONL格式.md §3.0](./02-字段与JSONL格式.md#30-ai-直出铁律最高优先级所有字段适用) |
| 脚本边界 | 脚本只允许做导入、删除、清类目、结构与禁词校验、生成 manifest；不允许产出任何内容字段 |

---

## 4. 幂等与替换

- 追加导入默认不删除旧数据；同标题是否重复由运营人工控制。
- 本地演示或重建数据使用 `--replace`。
- 生产环境不建议物理删除，优先通过管理端下架或软删。
- 文章 code 由 DB 生成，因此同一 JSON 在不同环境导入后 code 可能不同；文件 slug 用于内容追踪，DB code 用于线上 URL。

## 5. 导入后校验 SQL

```sql
select c.code, c.name_i18n->>'zh' as category, count(a.id) as articles
from zhiyu.china_categories c
left join zhiyu.china_articles a on a.category_id = c.id and a.deleted_at is null
group by c.code, c.name_i18n
order by c.code;

select a.title_i18n->>'zh' as title, count(s.id) as sentences, a.status
from zhiyu.china_articles a
join zhiyu.china_sentences s on s.article_id = a.id and s.deleted_at is null
where a.deleted_at is null
group by a.id
order by a.created_at desc;
```
