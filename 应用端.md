# 应用端 PRD：AI认知镜 V2.0（单人可交付版）

> 版本说明：本文档为独立开发者可在 6-8 周内交付的 MVP 版本。
> 删除了所有 WebGL 3D 渲染、空间音频、12 先知体系、社区广场、解压游戏等模块。
> 保留核心商业闭环：**付费AI认知报告 + 单级分销裂变**。

---

## 一、产品定位

**产品名称（对外）：** 内观（暂定，合规友好）
**产品名称（开发内部）：** AI认知镜

**一句话定位：** 描述你的困境，AI为你生成一份深度认知分析报告，帮你看清问题本质。

**变现方式：** 单次付费 ¥28.8，先付后看。

**分销方式：** 单级邀请分销，被邀请者付费后邀请人得 ¥8.64（30%）。

**合规定位：** 本产品为 AI 辅助认知工具，所有内容仅供参考，不构成专业建议。

---

## 二、技术栈

**前端（H5 Web）：** React + TypeScript + Tailwind CSS + Framer Motion
- 部署为 SPA 静态页面，由 Nginx 托管
- 用户通过移动端浏览器访问（支持微信内置浏览器 / Safari / Chrome）
- **不做微信小程序**（MVP 阶段），后期 App 通过 WebView 内嵌或 React Native 重构

**后端：** FastAPI（Python，全异步 async/await）

**数据库 & 认证：** Supabase（PostgreSQL + Supabase Auth，通过 MCP 直接操作）

**AI 调用：** Dify 工作流编排（调用 Claude/Deepseek 等，通过 Dify REST API）

**支付：**
- 微信支付 H5（MWEB）：适用于移动端非微信浏览器
- 微信支付 JSAPI：适用于微信内置浏览器（需公众号，优先推荐注册）
- 支付宝手机网站支付 H5

**短信服务：** 阿里云 SMS 或腾讯云 SMS（需提前注册，模板审核 1-3 天）

**部署：** Docker + Nginx（现有方案不变）

### 3.5 数据流 4：佣金自动结算

```
FastAPI（在订单完成后立即触发，同一事务内）
   │
   │  查询 orders.user_id = 买家
   │  查询 users.invited_by = 邀请人（可能为空）
   │
   ├──[自购返佣] INSERT commissions
   │  {beneficiary_id: 买家, order_id, type:'self_cashback',
   │   amount: 28.8 × 0.3 = 8.64, ratio: 0.3, status:'settled'}
   │
   ├──[推荐佣金] 若 invited_by 不为空：
   │  INSERT commissions
   │  {beneficiary_id: 邀请人, order_id, type:'referral',
   │   amount: 8.64, ratio: 0.3, status:'settled'}
   │
   ├──[更新钱包余额] UPDATE wallets
   │  SET balance += 8.64
   │  WHERE user_id = 买家
   │
   └──[若有邀请人] UPDATE wallets
      SET balance += 8.64,
          total_earned += 8.64
      WHERE user_id = 邀请人

原子性保证：以上操作放在同一个数据库事务（Supabase RPC 或 psycopg3 transaction）
若任何一步失败 → 整个事务回滚 → orders 状态仍为 'generating'，等待重试
```

### 3.10 关键安全与一致性规则

**JWT 验签（FastAPI 本地验签，不回调 Supabase）：**
```python
# 每次请求 O(1) 验证，不发 HTTP 请求
decoded = jwt.decode(
    token,
    settings.SUPABASE_JWT_SECRET,
    algorithms=["HS256"],
    audience="authenticated"
)
user_id = decoded["sub"]  # uuid
```

**Admin JWT（独立签发，与用户 JWT 完全隔离）：**
```python
# 登录时签发，role claim 标记
payload = {"sub": "admin", "role": "admin", "exp": now + 8h}
token = jwt.encode(payload, settings.ADMIN_JWT_SECRET, algorithm="HS256")
# 所有 /admin/* 接口验证 role == "admin"
```

**支付 Webhook 签名验证（必须，否则任何人可伪造支付成功）：**
```python
# 微信支付 v3
from wechatpayv3 import WeChatPay
result = wxpay.decrypt_callback(headers, body)  # 库处理签名验证

# 支付宝
from alipay import AliPay
verified = alipay.verify(params, sign)
```

**数据库 RLS 策略（每张表都必须启用）：**
```sql
-- users: 只能读自己
CREATE POLICY "users_self_only" ON users
  FOR ALL USING (auth.uid() = id);

-- orders: 只能读/写自己的订单
CREATE POLICY "orders_self_only" ON orders
  FOR ALL USING (auth.uid() = user_id);

-- commissions: 只能读自己作为受益人的佣金
CREATE POLICY "commissions_self_only" ON commissions
  FOR SELECT USING (auth.uid() = beneficiary_id);

-- wallets: 只能读自己
CREATE POLICY "wallets_self_only" ON wallets
  FOR SELECT USING (auth.uid() = user_id);

-- 管理端接口: 通过 Admin JWT (service_role key) 绕过 RLS
```

## 四、全局UI规范

### 3.1 视觉基调

**整体风格：** 极简暗色 + 鎏金点缀。高级感来自克制，不来自堆砌特效。

**主色系：**
- 背景：`#0A0A0F`（近黑）
- 面板：`rgba(255,255,255,0.06)` + `border: 1px solid rgba(255,255,255,0.1)` 实现毛玻璃感
- 强调色：`#C9A84C`（鎏金）
- 文字主色：`#F0EDE6`
- 文字次色：`rgba(240,237,230,0.5)`

**毛玻璃实现（CSS，无 WebGL）：**
```css
backdrop-filter: blur(12px);
background: rgba(255, 255, 255, 0.06);
border: 1px solid rgba(255, 255, 255, 0.1);
border-radius: 16px;
```

**背景：** 固定的 CSS Mesh Gradient 渐变，不做实时动态流体计算。暗金色与深黑色系。

**字体：** 系统默认无衬线字体（`-apple-system, BlinkMacSystemFont, 'PingFang SC'`）。不引入外部字体，减少加载时间。

**动效：** 使用 Framer Motion 做页面切换淡入淡出 + 按钮点击缩放。不做复杂物理动效。

**帧率要求：** 所有页面在中端机（骁龙 870 级别）上保证 60fps。只做单一渲染等级，不做 S/A/B 降级（MVP 阶段不必要）。

### 3.2 按钮规范

所有可点击元素统一为 **带圆角的矩形水滴按钮**，圆角 `border-radius: 12px`。

主要操作按钮：鎏金渐变背景 + 深色文字。
次要操作：透明背景 + 细边框。

## 五、页面详细说明

### 5.1 登录 / 注册页

**进入时机：** 用户未登录时，点击「开始」后跳转。

**UI：**
- 全屏暗色背景 + 中央毛玻璃面板
- 面板内：Logo（文字版，不做复杂图形）+ 输入区
- 输入框仅保留底部下划线样式，聚焦时泛起金色光晕

**登录方式（MVP 只做以下两种）：**

1. **手机号 + 短信验证码**
   - 手机号输入框 → 发送验证码按钮 → 验证码输入框 → 登录/注册
   - 新手机号：自动创建账号
   - 已注册手机号：直接登录

2. **微信小程序端：微信一键登录**
   - 调用微信 `wx.login` + `getPhoneNumber` 组件
   - 后端用 code 换取 openid + 手机号
   - 与手机号账号自动合并（同一手机号视为同一账号）

**用户唯一标识：** Supabase 的 `uuid`，不用手机号做主键。

**验证码防刷（必须上线前实现）：**
- 同一手机号：60秒内只能发 1 次
- 同一 IP：1小时内最多发 10 次
- 超限返回：「发送过于频繁，请稍后重试」
- 后端用 FastAPI 进程内字典实现内存缓存（MVP 单容器部署足够），不依赖第三方 Redis

**JWT 有效期：** 登录成功后，Token 有效期 30 天（不做1年长效，降低安全风险）。

### 5.9 邀请落地页

**URL：** `/invite/{邀请码}`

**页面内容：**

- 「立即体验」按钮 → 跳转注册页
- 注册时后端自动记录 `inviter_code`，绑定邀请关系

**绑定规则：**
- 一个用户只能被一个邀请人绑定（先到先得）
- 已注册用户点击邀请链接不重新绑定

**佣金触发规则（确认版）：**

1. **自购返佣**：每个用户自己购买报告，自己获得 30% 返佣（¥8.64）进入可提现余额
2. **推荐佣金**：用户的直接被邀请人每次购买，邀请人获得 30% 佣金（¥8.64）
3. **双重叠加**：若用户 A 邀请了用户 B，用户 B 购买时：A 得 ¥8.64（推荐佣金）+ B 也得 ¥8.64（自购返佣）
4. **触发时机**：每笔订单支付成功后立即结算，无延迟
5. **动态比例**：MVP 阶段固定 30%，Phase 2 根据累计消费/邀请量实现阶梯提升（待 Phase 2 设计）

**每笔 ¥28.8 订单的佣金分配：**
- 无邀请人：买家自得 ¥8.64，平台净得 ¥20.16（70% 毛利）
- 有邀请人：买家自得 ¥8.64 + 邀请人得 ¥8.64，平台净得 ¥11.52（40% 毛利）

## 十、商业模型与盈利结构

### 10.1 单笔订单利润拆解

每笔 ¥28.8 订单的收入拆解框架（**注意：AI 调用成本数字需根据实际 Dify 使用数据校准，以下为估算框架**）：

| 项目 | 金额 | 说明 |
|------|------|------|
| 用户支付 | ¥28.8 | 含税价格 |
| 支付通道手续费 | ~¥0.58 | 约 2%（微信/支付宝） |
| AI 调用成本（估算） | ~¥0.5~1.5 | 根据 prompt 长度和模型选择浮动，上线后实测 |
| **无邀请时净利润** | **~¥26.7~27.7** | **净利润率约 93%** |
| 有邀请时佣金支出 | ¥8.64 | 30% 给邀请人 |
| **有邀请时净利润** | **~¥18.1~19.1** | **净利润率约 63-66%** |

**关键结论：** 即使 100% 的订单都来自邀请分销，净利润率仍超过 60%。这是非常健康的数字内容产品利润结构。

**需要上线后验证的数据：**
- 实际 AI 调用成本（去 Dify 后台看 token 消耗）
- 自然流量 vs 邀请流量的比例
- 用户复购率（同一用户第二次付费）

### 10.2 盈亏平衡测算

| 场景 | 每月订单数 | 月收入（估算） |
|------|----------|-------------|
| 最小验证 | 50 单 | ~¥900-1,000（扣除所有成本后） |
| 初步可行 | 200 单 | ~¥3,600-4,000 |
| 良性运营 | 500 单 | ~¥9,000-10,000 |

500 单/月 = 每天约 17 单。通过分销裂变驱动是完全可达的目标。

## 十一、产品演进路线

V2.0 MVP 验证核心变现逻辑后，按以下方向扩展：

### Phase 2：会员订阅制（验证后 3-6 个月）

**目标：** 在单次付费基础上叠加订阅收入，提升用户 LTV。

**方案：** 新增 ¥68/月 会员方案
- 会员每月可生成 3 份报告（折合每份约 ¥22.7，比单次便宜）
- 会员享有优先排队（生成队列优先处理）
- 会员报告支持追问功能（对报告内容进行一次追问）

**实现说明：**
- 数据库新增 `subscriptions` 表，记录会员有效期
- 订单创建接口判断用户会员状态，扣减月度使用次数
- 接入微信/支付宝「周期扣款」或改为手动续费（MVP 先做手动，降低实现复杂度）

**触发时机：** 月活跃付费用户超过 100 人后，再开发订阅模块。不要提前投入。

### Phase 3：实物产品电商（长期方向，12 个月以后）

**背景：** 参考海外 Astrotalk 等平台，在建立用户信任后，以「工具书/灵修类实物商品」作为高毛利收入补充。

**方向：**
- 精选少量高毛利商品（如：认知笔记本、冥想辅助品）
- 用内容心智驱动购买，不做纯电商流量逻辑
- 与现有用户基础结合，不开新域名，在「我的」页面内增加商城入口

**注意：** Phase 3 需要供应链和客服支持，复杂度远超纯数字产品。不要在 MVP 阶段考虑，只作为长期方向记录。
