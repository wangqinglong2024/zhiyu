## 一、触发场景

### 1.1 触发条件

| 场景 | 触发行为 | 前置条件 |
|------|---------|---------|
| Level 地图点击锁定 Level | 用户在课程首页点击状态为 🔒（锁定）的 L4-L12 Level 节点 | 该 Level 未购买 |
| Level 详情页返回地图后点击下一级 | 用户完成当前免费 Level（L3）综合考核后，点击 L4 | L4 未购买 |
| 入学测试推荐付费 Level | 测试结果推荐 L4+ 且用户点击"开始学习 Level X" | 该 Level 未购买 |
| 个人中心课程购买 | 用户在个人中心 → 课程购买列表中点击未购买的 Level | 该 Level 未购买 |

### 1.2 不触发的场景

- 点击 L1-L3（免费级别）→ 直接进入 Level 详情页
- 点击已购买的 L4-L12 → 直接进入 Level 详情页
- 已登录但未完成前序 Level 的已购买 Level → 直接进入 Level 详情页（但单元解锁受限于前序进度）

## 二、弹窗布局

### 2.1 弹窗形态

- **底部弹出面板**（Bottom Sheet），从屏幕底部向上滑入
- 圆角：顶部 32px（`--radius-3xl`），底部无圆角（贴底）
- 背景：毛玻璃材质（`.glass-card`）
- 遮罩层：页面背景半透明黑色遮罩（`rgba(0,0,0,0.5)`），点击遮罩可关闭弹窗
- 弹窗高度：自适应内容，最大不超过屏幕 85%
- 滑入动画：300ms，Decelerate 缓动

### 2.4 Paddle 支付流程

用户点击"购买 $6"按钮后的流程：

```
① 按钮变为 Loading 态（转圈 + 文字"正在准备支付…"）
② 前端调用后端创建 Paddle Checkout Session
③ 打开 Paddle 托管支付页面（Overlay 模式，覆盖在当前页面上方）
④ 用户在 Paddle 页面完成支付（输入卡号/选择支付方式/确认）
⑤ 支付完成后 Paddle 关闭 Overlay → Paddle Webhook 通知后端
⑥ 后端更新用户课程权限 → 通知前端
⑦ 前端执行支付成功处理（详见 §三）
```

## 三、支付成功后行为

| 步骤 | 行为 | 时机 |
|------|------|------|
| 1 | 关闭付费墙弹窗（包括二次确认弹窗） | 收到后端确认后立即执行 |
| 2 | 课程首页 Level 地图中对应 Level 节点状态更新 | 从 🔒（锁定）→ ⚪（已解锁待学）或 🔵（已购买待学） |
| 3 | 顶部 Toast 通知 | "购买成功，开始学习吧！"（绿色 Toast，2s 自动消失），多语言（EN: "Purchase successful, start learning!" / VI: "Mua thành công, bắt đầu học thôi!"） |
| 4 | 自动滚动到已解锁的 Level 节点 | 地图滚动到该 Level 节点位置居中，500ms 动画 |

### 状态更新逻辑

- 若用户前序 Level 已完成综合考核 → 新购 Level 变为 ⚪（已解锁待学），可直接进入
- 若用户前序 Level 尚未完成 → 新购 Level 变为 🔵（已购买待学），需先完成前序 Level

## 四、支付失败处理

### 4.1 Paddle 支付失败

| 场景 | 处理 |
|------|------|
| 用户主动取消支付 | Paddle Overlay 关闭 → 付费墙弹窗保持打开 → 无额外提示（用户主动取消无需提醒） |
| 支付方式被拒（卡余额不足等） | Paddle 页面内展示错误提示 → 用户可在 Paddle 内换卡重试 → 若用户关闭 Paddle → 付费墙弹窗保持打开 |
| 网络超时 | 按钮恢复可用态 → 红色 Toast "支付请求超时，请检查网络后重试"（多语言），3s 需手动关闭 |
| Paddle Webhook 延迟 | 按钮显示 Loading 最长 15s → 超时后提示"支付处理中，请稍后刷新查看"（多语言）→ 后端异步处理完成后推送通知 |

### 4.2 知语币兑换失败

| 场景 | 处理 |
|------|------|
| 兑换过程中余额变动（低于 600） | 后端校验失败 → 红色 Toast "兑换失败，知语币余额不足"（多语言）→ 弹窗余额数字刷新 |
| 网络错误 | 红色 Toast "兑换失败，请检查网络后重试"（多语言）→ 按钮恢复可用态 |
