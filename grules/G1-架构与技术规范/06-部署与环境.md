# 06 · 部署与环境

> 引用：[00-index.md](./00-index.md) · [env.md](../../env.md)
>
> **核心约束**：Docker-only，**唯一一个 dev 环境**用于自动化测试，**禁止多环境**。
> dev 通过 `IP:端口` 直接访问，**不挂反向代理**；生产部署由用户自行处理（用户选 Nginx + 自配 HTTPS）。
>
> **代码根**：所有开发代码与 Docker 配置位于 `/opt/projects/zhiyu/system/`；`docker compose`、`pnpm`、`supabase` 命令 cwd 均为 `system/`。本文中的 `docker/`、`apps/`、`supabase/`、`scripts/` 等路径均隐含 `system/` 前缀。

## 一、Docker 拓扑（唯一 dev 环境）

```
┌──────────── docker compose（file: docker/compose.yaml）─────────────┐
│                                                                     │
│  web-app           :3100   →  应用端前端（Vite preview / dev）       │
│  web-admin         :4100   →  管理后台前端                           │
│  api-app           :8100   →  应用端后端（Hono + @hono/node-server） │
│  api-admin         :9100   →  管理后台后端                           │
│  api-app-worker    —       →  api-app 的 BullMQ worker（同镜像）     │
│  redis             :6379   →  缓存 + BullMQ（zhiyu 专用）            │
│                                                                     │
│  ── 复用主机 supabase_default 外部网络（不再自建）──                  │
│  supabase-db       :5432   →  supabase/postgres:15.x                 │
│                                  （pgvector 内置，AI 场景按需启用）   │
│  supabase-kong     :8000   →  PostgREST/Auth/Storage/Realtime 总入口 │
│  supabase-studio   :8000/  →  通过 kong 进入                         │
│  supabase-edge-fn  内网    →  Edge Functions（Deno）                 │
│                                                                     │
│  （无反向代理；前端直接连 IP:8000、IP:8100、IP:9100）                │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.1 端口策略

| 用途 | 端口 | 说明 |
|------|------|------|
| 应用端前端 | **3100** | 用户指定 |
| 应用端后端 | **8100** | 用户指定（Hono）|
| 管理后台前端 | **4100** | 用户指定 |
| 管理后台后端 | **9100** | 用户指定（Hono）|
| Supabase Kong（API 总入口）| 8000 | 前端浏览器直连 |
| Supabase Studio | 54323 | 仅 dev 调试 |
| Supabase Edge Functions | 54321 | 鉴权 / 回调 / 轻量接口 |
| Postgres | 5432 | 仅 host 调试用，业务走 docker 网络 |
| Redis | 6379 | 同上 |

> **占用即释放**：`scripts/dev/free-ports.sh` 在 `docker compose up` 前自动 `kill` 占用 `3100/4100/8100/9100` 的进程；CI/dev 强制启用，不询问。

### 1.2 启动命令

```bash
# 一键启动唯一 dev 环境
pnpm dev               # = scripts/dev/free-ports.sh && docker compose up --build

# 跑测试
docker compose run --rm api-app   pnpm test
docker compose run --rm api-admin pnpm test
docker compose run --rm web-app   pnpm test
docker compose run --rm web-admin pnpm test

# 数据库迁移 / 类型生成
pnpm supabase migration new <name>
pnpm supabase db reset
pnpm supabase gen types typescript --local > packages/supabase-client/src/types/database.ts
```

| 镜像 | 基础镜像 | 阶段 |
|------|---------|------|

## 三、环境变量

### 3.1 来源与优先级

`.env`（dev 默认）  ←  `docker/env/.env.dev`（仓库提交，**仅占位**）  ←  本机 `docker/env/.env.local`（不提交，覆盖密钥）

### 3.2 必备变量

| 变量 | 示例 | 说明 |
|------|------|------|
| `APP_ENV` | `dev` | 唯一环境 |
| `APP_NAME` | `zhiyu` | 项目名 |
| `DATABASE_URL` | `postgres://postgres:postgres@supabase-db:5432/postgres` | 仅供脚本/RPC 测试，业务走 supabase-js |
| `REDIS_URL` | `redis://redis:6379/0` | 缓存 + BullMQ |
| `SUPABASE_URL` | `http://supabase-kong:8000` | 后端调用 |
| `SUPABASE_ANON_KEY` | `…` | 前端公开 |
| `SUPABASE_SERVICE_ROLE_KEY` | `…` | 后端绕过 RLS |
| `JWT_SECRET` | 32+ 字符 | 与 Supabase 一致 |
| `VITE_API_BASE_APP` | `http://localhost:8100/api/v1` | 应用端前端 |
| `VITE_API_BASE_ADMIN` | `http://localhost:9100/admin/v1` | 管理后台前端 |
| `VITE_SUPABASE_URL` | `http://localhost:8000` | 前端浏览器直连 Kong |
| `VITE_SUPABASE_ANON_KEY` | `…` | 同上 |
| `LOG_LEVEL` | `debug` | dev 默认 |
| `RATE_LIMIT_RPM` | `60` | 限流默认 |

### 3.3 第三方 / AI Key（缺则 mock，不阻塞）

> `docker/env/.env.example` 必须列出**全部**变量与说明，缺失时由对应 adapter fallback 到 mock。

| 变量 | 用途 | 缺失行为 |
|------|------|---------|
| `ANTHROPIC_API_KEY` | Claude（主创作模型）| `LLMAdapter` → mock fixture |
| `DEEPSEEK_API_KEY` | DeepSeek（副 / TTS）| 同上 |
| `OPENAI_API_KEY` | Embedding / 备用 | 走 hash 占位 embedding |
| `LANGSMITH_API_KEY` | LangGraph 评估（可选）| 跳过上报 |
| `SMTP_*` / `RESEND_API_KEY` | 邮件 | console + 落 `outbox` 表 |
| `GOOGLE_OAUTH_CLIENT_ID` / `GOOGLE_OAUTH_CLIENT_SECRET` | Google 登录 | **mock 验证通过**，生成假用户 |
| `PADDLE_API_KEY` / `PADDLE_PUBLIC_KEY` / `PADDLE_WEBHOOK_SECRET` | Paddle 支付 | **全量 mock**：mock 下单 + mock 回调 |

> **新加 Key 流程**：在 `docker/env/.env.example` 增补占位 + 在 `docker/env/README.md` 写明申请方式 → adapter 增加 fallback 实现 → 提 PR。

## 四、Supabase 自托管（本地化）

> **不使用 supabase.com 云服务**。所有 Supabase 组件由 `docker/compose.yaml` 拉起。

1. 镜像：参照官方 `supabase/docker/docker-compose.yml`，按需裁剪到 `docker/compose.yaml` 同一文件。
2. Edge Functions：由 `supabase functions serve --env-file ./docker/env/.env.dev` 在独立容器（Deno）启动；监听 54321。
3. 初始化：
   - `docker/postgres/init/00-extensions.sql` 启用 `uuid-ossp`、`pgcrypto`、`vector`、`pg_trgm`；
   - `pnpm supabase db reset` 执行 `supabase/migrations/*.sql` + `supabase/seed.sql`；
   - 失败立即退出，让 compose 标记不健康。
4. Studio 仅 dev 内使用，**不暴露公网**；admin 后台不依赖 Studio。

## 六、日志与可观测（dev 默认）

- 容器 stdout JSON → `docker logs`；不挂载到宿主，体积由 Docker logging driver 控制（`max-size=20m, max-file=5`）。
- 错误聚合（可选）：本地 Loki + Grafana 容器，默认关闭，由 `compose.yaml` 的 `profiles: [observability]` 提供。

## 七、备份与恢复

- dev 环境**默认不开启外部备份**（用户偏好）。
- 提供 `scripts/db/dump.sh` / `restore.sh` 供本地手动使用。

---

## 八、CI（容器内自动化）

- 流水线（如有）只依赖 Docker：`docker compose up -d` → `docker compose run --rm <svc> pnpm <task>` → `docker compose down -v`。
- **不集成 GitHub Actions / 第三方 SaaS**（user memory `zhiyu-docker-policy.md`）。

---

## 九、生产部署（仅备注，非本仓库交付）

- 反向代理：用户指定 **Nginx**；HTTPS 证书与多域名拆分由用户自行维护。
- dev 镜像与生产镜像构建参数一致，仅 `.env.local` 不同。
- 前端无需 SSR 服务，Nginx 直接 `try_files $uri /index.html` 即可承载 SPA。
