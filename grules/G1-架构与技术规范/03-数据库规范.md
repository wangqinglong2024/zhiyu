## 一、命名规则

| 对象 | 规则 | 示例 |
|------|------|------|
| 表名 | `snake_case` 复数名词 | `users`、`course_lessons`、`game_sessions` |
| 关联表 | `表a_表b`（按字典序）| `users_roles` |
| 字段名 | `snake_case` | `created_at`、`display_name` |
| 主键 | `id`，类型 `uuid`（`gen_random_uuid()`）| `id uuid` |
| 外键 | `<引用表单数>_id` | `user_id`、`course_id` |
| 索引 | `idx_<表>_<字段...>` | `idx_users_email` |
| 唯一索引 | `uq_<表>_<字段...>` | `uq_users_email` |
| 外键约束 | `fk_<表>_<字段>` | `fk_orders_user_id` |
| RPC 函数 | `fn_<动词>_<对象>` | `fn_purchase_course`、`fn_calc_user_level` |
| 视图 | `v_<对象>` | `v_user_summary` |
| 触发器 | `tg_<表>_<时机>_<动作>` | `tg_users_before_update_set_updated_at` |
| 枚举值 | `lower_snake` 字符串 | `'pending'`、`'in_progress'` |
| 布尔字段 | `is_xxx` / `has_xxx` / `can_xxx` | `is_active`、`has_paid` |
| 时间戳字段 | `xxx_at`，类型 `timestamptz` | `published_at` |

## 二、通用字段（每张表都有）

| 字段名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `id` | `uuid` | `gen_random_uuid()` | 主键 |
| `created_at` | `timestamptz` | `now()` | 创建时间 |
| `updated_at` | `timestamptz` | `now()` | 更新时间，由触发器自动维护 |
| `deleted_at` | `timestamptz` | `null` | 软删除标记，仅"用户可生成数据"表需要 |

> **`updated_at` 触发器**：在 `supabase/migrations/0000_helpers.sql` 中定义统一函数 `set_updated_at()`，每张表迁移末尾绑定。

---

## 三、软删除策略

- **采用软删除的表**：所有"用户可生成且可能恢复"的数据。
  - 如：`users`、`courses`（用户自建）、`novels`、`comments`、`orders`。
- **不采用软删除（直接物理删除）**：
  - 系统配置表、缓存表、日志表、临时会话（`game_sessions`）、排行榜快照、消息已读标记。
- **查询默认**：`packages/supabase-client/src/helpers/soft-delete.ts` 提供 `selectActive(table)` 包装，默认拼 `.is('deleted_at', null)`；写删除接口必须显式调用 `softDelete(table, id)`。
- **唯一索引**：所有 `unique` 索引必须 `WHERE deleted_at IS NULL`，避免软删后无法复用同名。

## 四、字段类型选用

| 业务场景 | 类型 | 说明 |
|---------|------|------|
| 标识符 | `uuid` | 全表统一 |
| 短文本 | `text` | Postgres `varchar(n)` 与 `text` 性能一致，统一 `text`，由 Zod 限长 |
| 长文本 / 富文本 | `text` | 配合应用层长度校验 |
| 金额 | `numeric(18,2)` | 不要用 `float` |
| 计数 | `bigint` | 防止 32 位溢出 |
| 标签数组 | `text[]` | 5 个元素以下；多则建关联表 |
| 半结构化数据 | `jsonb` | 写入注意见 §九 |
| 多语言文案 | `jsonb`，结构 `{zh,en,vi,th,id}` | 与 i18n 5 语言一致；缺省回退 `en` |
| 向量（**仅 AI 场景**） | `vector(1536)` | 与默认 embedding 维度一致；使用前必须先在该迁移内 `create extension if not exists vector;`，**禁止在非 AI 业务表使用 vector 类型** |
| 枚举 | `text` + `CHECK` 约束 | 不用 PG enum 类型，避免迁移痛点 |

## 五、索引与约束规范

1. 外键字段**必须**建索引。
2. 列表页排序字段、筛选字段建组合索引（最左前缀原则）。
3. 全文检索字段使用 `tsvector` 生成列 + GIN 索引。
4. JSONB 经常查询的路径建表达式索引。
5. 所有外键声明 `ON DELETE` 行为：
   - 用户级数据：`ON DELETE CASCADE`；
   - 系统/字典引用：`ON DELETE RESTRICT`；
   - 弱引用（统计/日志）：`ON DELETE SET NULL`。

---

## 六、Schema 与迁移（Supabase CLI）

1. 新建迁移：`pnpm supabase migration new <verb>_<object>` → 在 `supabase/migrations/` 生成空 SQL。
2. 编写纯 SQL；Schema 改动只走此路径。
3. 本地应用：`pnpm supabase db reset`（dev 环境随时全量重建）。
4. 类型同步：`pnpm supabase gen types typescript --local > packages/supabase-client/src/types/database.ts`，提交到仓库。
5. **禁止手改已合入的迁移**；修改字段必须新增迁移。
6. 数据回填脚本放 `scripts/backfill/`，TypeScript 编写，`tsx` 执行；不与 schema 迁移混在一起。
7. 复杂查询/事务封装为 **Postgres 函数（RPC）**，由后端 `supabase.rpc(...)` 调用，避免在应用层堆 SQL 字符串。

---

## 七、RLS（Row Level Security）

- **C 端可读写表**：必须开启 RLS，Policy 至少覆盖 `select / insert / update / delete` 四种操作。
- **后台专用表**：开启 RLS 并仅放行 `service_role`。
- Policy 命名：`<表>_<操作>_<角色>`，如 `orders_select_owner`。
- 测试要求：`supabase/tests/rls/` 必须为每张开启 RLS 的表写至少 1 条正例 + 1 条反例（`pgTAP` 或 `pnpm test:rls` 自定义脚本）。
- **后端 service-role 客户端绕过 RLS**：仅 `apps/api-*` 与 Edge Functions 可使用；前端禁止接触。

## 八、多语言文案存储

- 文案 jsonb 结构：

  ```json
  { "zh": "你好", "en": "Hello", "vi": "Xin chào", "th": "สวัสดี", "id": "Halo" }
  ```

- 应用层读取：按用户当前 locale 取值；缺失依次回退 `en` → `zh`。
- 校验：Zod schema 强制至少包含 `zh`、`en`。

---

## 九、JSONB 使用注意事项

- 通过 **supabase-js / PostgREST** 直接 `insert({ col: obj })` 是**安全**的（走 JSON body，PostgREST 自动识别 jsonb 列，不会双重编码）。
- 通过 **RPC** 传 jsonb 参数时：
  - SQL 函数签名必须显式 `arg jsonb`；
  - JS 端调用：`supabase.rpc('fn_x', { payload: obj })`，**禁止**先 `JSON.stringify` 再传，否则 PostgREST 当成字符串入参，导致 `jsonb_typeof = 'string'`。
- 历史教训（user memory `drizzle-jsonb.md`）：本项目**不使用 Drizzle**，相关坑作废；新规则以本节为准。

## 十、缓存表与滚动窗口

- 滚动窗口缓存表 PK `period_start` 必须为**稳定桶键**（如 `date_trunc('hour', now())`），不得使用 `now() - Δ`，否则 `ON CONFLICT` 永不命中。
