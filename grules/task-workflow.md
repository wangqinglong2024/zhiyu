# AI 驱动的任务拆解与开发流程 (Task Workflow)
> **版本**: v1.2 | **最后更新**: 2025-07-16
>
> **本文件是整套规范的"发动机"**。
> 它定义了：你给 AI 一份产品需求 → AI 如何拆解成可执行任务 → 如何逐个开发 → 如何验收的完整闭环。

---

## 一、总体工作流（6 步循环）

```
┌─────────────┐
│  第 0 步     │  你提供原始需求 / Stitch 原型
│  需求优化    │  → AI 按 product-design.md 优化 PRD + 审查 Stitch 原型
│  + 原型审查  │  → 通过质量门禁后进入下一步
└──────┬──────┘
       ▼
┌─────────────┐
│  第 1 步     │  优化后的【产品需求文档】（PRD）
│  产品需求    │  → 放在 product/ 目录下（已通过质量门禁）
└──────┬──────┘
       ▼
┌─────────────┐
│  第 2 步     │  AI 读取 PRD + grules/ 下所有规范文件
│  需求分析    │  → 输出【功能模块清单】+ 数据库表设计 + API 清单
└──────┬──────┘
       ▼
┌─────────────┐
│  第 3 步     │  AI 将功能模块拆解为【开发任务列表】放到 tasks/下
│  任务拆解    │  → 每个任务粒度：1 个开发者 2~4 小时能完成
└──────┬──────┘
       ▼
┌─────────────┐
│  第 4 步     │  你说"开始第 N 个任务"
│  逐任务开发  │  → AI 按规范生成代码、自测、交付
└──────┬──────┘
       ▼
┌─────────────┐
│  第 5 步     │  所有任务完成后
│  集成验收    │  → AI 做端到端联调测试
└─────────────┘
```

---

## 二、第 0 步：需求优化与原型审查（新增前置阶段）

> **详细规则见 `product-design.md`**。此处仅说明衔接关系。

在用户提供原始需求（大白话或 Stitch 原型）后，AI 先进行：
1. **PRD 优化**：按 8 维度质量标准（用户故事、边界条件、优先级、验收标准、数据流向、非功能需求、情感化设计、商业指标）补全优化
2. **Stitch 原型审查**（如有原型）：对照 PRD + rules.md 的 UI 规范逐屏审查
3. **通过门禁**：PRD 达标 + 原型审查通过后，才进入第 1 步

标准对话模板：
```
我有一个新产品想法。
请先阅读 grules/product-design.md，
然后帮我把需求优化成标准 PRD。
如果 Stitch 中已有原型，请一并审查。
```

## 三、第 1 步：产品需求文档（PRD）模板

你写需求时不需要懂技术，只需要用大白话描述。将 PRD 保存到 `product/` 目录。
AI 会帮你翻译成技术方案。

### 你需要写清楚的内容
```markdown
# 产品名称：XXX

## 产品简介
一句话说清楚这个产品是干嘛的。

## 目标用户
谁会用这个产品？

## 核心功能列表
用编号列出你能想到的所有功能，越详细越好：

1. 用户注册登录（手机号 + 验证码）
2. 用户可以创建自己的 XX
3. 用户可以浏览其他人的 XX
4. 点赞 / 收藏功能
5. 管理员后台可以查看所有 XX
6. ...

## 每个功能的细节描述（可选但推荐）
### 功能 1：用户注册登录
- 支持手机号 + 验证码登录
- 首次登录自动注册
- 登录后跳转到首页
- ...

### 功能 2：...
（以此类推）

## 页面列表（如果你心里有数的话）
- 首页
- 登录页
- 详情页
- 个人中心
- ...

## 特殊要求（可选）
- 需要微信支付
- 需要对接某个第三方 API
- 必须支持手机端访问
- ...
```

## 四、第 2 步：AI 需求分析输出格式

当你把 PRD 给 AI 后，要求 AI 先输出以下分析文档（而不是直接写代码）：

### 你可以直接复制这段话发给 AI：
```
请阅读 product/ 目录里的需求文档，以及 grules/ 目录下的所有规范
文件（rules.md、coding-standards.md、project-structure.md、api-design.md、qa-testing.md）。
然后帮我输出：

1. 【功能模块清单】：把需求拆分成独立的功能模块，列出每个模块包含哪些子功能
2. 【数据库表设计】：设计所有数据库表，包含字段名、类型、约束、表间关系
3. 【API 接口清单】：列出所有需要开发的 API 端点，包含方法、URL、请求/响应格式
4. 【页面清单】：列出所有前端页面 + 每个页面用到哪些组件

先不要写代码，我确认方案后再开始开发。
```

## 五、第 3 步：任务拆解规则

### 拆解原则
1. **粒度适中**：每个任务大约对应 2~4 小时的开发量（AI 可能几分钟就完成）
2. **依赖顺序**：先基础设施 → 再后端 → 然后前端 → 最后联调
3. **可独立验证**：每个任务完成后都能单独运行/测试
4. **前后端分离**：同一个功能的后端 API 和前端 UI 是独立的两个任务

### 标准任务拆解顺序

```
阶段 0：项目初始化（只做一次）
  T-0.1  生成 Docker + 项目骨架
  T-0.2  后端基础设施（config, auth, exceptions, middleware, main.py）
  T-0.3  前端基础设施（lib/, providers/, router/, 全局样式）
  T-0.4  数据库初始迁移 + RLS 策略

阶段 1：核心功能后端
  T-1.1  [模块A] 数据库建表 + 迁移文件
  T-1.2  [模块A] Pydantic 模型 + Repository 层
  T-1.3  [模块A] Service 层 + Router 层
  T-1.4  [模块B] 数据库建表 + 迁移文件
  T-1.5  [模块B] Pydantic 模型 + Repository 层
  T-1.6  [模块B] Service 层 + Router 层
  ...

阶段 2：核心功能前端
  T-2.1  [模块A] 页面 + 组件 + Hook + Service
  T-2.2  [模块B] 页面 + 组件 + Hook + Service
  ...

阶段 3：次要功能
  T-3.x  （按同样的 后端→前端 节奏）

阶段 4：集成与优化
  T-4.1  端到端联调测试
  T-4.2  错误处理完善 + Loading 状态
  T-4.3  响应式适配 + 暗色模式检查
  T-4.4  性能优化（懒加载、缓存策略）
```

### 你可以直接发给 AI 的话：
```
方案我确认了，请按照 grules/task-workflow.md 里的拆解规则，
帮我把所有功能拆解成开发任务列表。

输出格式：
任务编号 | 任务标题 | 涉及文件 | 前置依赖 | 验收标准
```

### AI 应输出的任务列表格式
```markdown
| 编号 | 任务标题 | 涉及文件 | 前置依赖 | 验收标准 |
|------|---------|---------|---------|---------|
| T-0.1 | 项目骨架 + Docker 配置 | docker-compose.yml, Dockerfile ×2, .env.example | 无 | docker compose up 能启动前后端容器 |
| T-0.2 | 后端基础设施 | app/core/*.py, app/main.py | T-0.1 | /docs 能打开 Swagger; /health 返回 200 |
| T-0.3 | 前端基础设施 | src/lib/*, src/styles/*, src/providers/* | T-0.1 | npm run dev 能打开页面，毛玻璃背景正常 |
| T-1.1 | 用户表 + RLS | migrations/xxx.sql | T-0.2 | 表创建成功，RLS 生效 |
| T-1.2 | 用户模型 + 数据访问 | models/user.py, repositories/user_repo.py | T-1.1 | 单元测试通过 |
| ... | ... | ... | ... | ... |
```

## 六、第 4 步：逐任务开发协议

### 你启动单个任务时，直接说：
```
开始任务 T-1.3：[任务标题]
```

### AI 在执行每个任务时必须遵守：

1. **先读后写**
   - 先检查前置依赖任务的代码是否存在
   - 先读取相关的现有文件，理解上下文
   - 然后再写新代码

2. **严格按规范**
   - 代码风格 → `coding-standards.md`
   - 注释与文档 → `documentation-standards.md`
   - UI/UE 设计 → `ui-design.md` + `rules.md` §一（CSS 参数）
   - 文件位置 → `project-structure.md`
   - API 设计 → `api-design.md`
   - 架构底线 → `rules.md`

3. **交付清单**
   每个任务完成后，AI 必须输出：
   ```
   ✅ 任务 T-X.X 完成
   
   【完成状态】：✅ 完成 / ⚠️ 完成但有顾虑 / 🚫 阻塞 / ❓ 需要上下文
   
   本次新增/修改的文件：
   - backend/app/routers/v1/users.py（新增）
   - backend/app/services/user_service.py（新增）
   - backend/app/models/user.py（修改：新增 UserUpdate 模型）
   
   验收方式：
   - 启动后端容器，访问 /docs
   - 测试 PATCH /api/v1/users/me 接口
   - 预期返回更新后的用户信息
   
   QA 测试结果：
   - Docker 服务状态：✅ 全部 Running
   - 冒烟测试：✅ 通过（控制台无 Error）
   - 功能测试：✅ 核心操作验证通过（附截图）
   - 回归影响：无（未改动已有功能）
   
   已知依赖：
   - 前端调用此 API 将在 T-2.1 完成
   ```

4. **自测（必须按 qa-testing.md 执行）**
   - 后端任务：`docker compose up -d --build backend`，确保无 import 错误，API 可调通
   - 前端任务：`docker compose up -d --build frontend`，确保 build 通过，页面可访问
   - 数据库任务：确保 SQL 语法正确，RLS 策略有效
   - 涉及 UI 的任务：必须通过 Browser MCP 截图验证
   - 所有任务：检查控制台无新增 Error

5. **任务失败回退流程**

   当任务开发过程中遇到不可解决的问题，按以下优先级处理：

   | 场景 | 操作 | 命令参考 |
   |------|------|---------|
   | Docker 构建失败 | 查日志 → 修复 → 重建 | `docker compose logs <service>` → 修复代码 → `docker compose up -d --build <service>` |
   | 代码变更破坏已有功能 | 撤回单个文件 | `git checkout -- <file>` |
   | 任务做到一半被阻塞 | 暂存进度，报告阻塞 | `git stash push -m "WIP T-X.X: 简述"` → 状态设为 🚫 阻塞 |
   | 变更彻底不可挽回 | 硬回退到上次提交 | `git reset --hard HEAD` （仅限尚未 push 的改动） |
   | 数据库 Migration 失败 | 回滚 SQL + 删除迁移文件 | 编写 `DOWN` 回滚语句执行 → 删除出错的 `.sql` 文件 → 重新生成 |

   **回退纪律**：
   - 每个任务开始前先 `git status` 确认工作区干净
   - 回退后必须重新跑 `docker compose up -d --build` 验证系统完整性
   - 同一问题尝试 3 次仍失败 → 停止尝试，报告 🚫 阻塞状态并附带完整错误日志

## 七、第 5 步：集成验收检查清单

所有任务完成后，用这个清单做最终检查：

```markdown
## 功能验收
- [ ] 所有 PRD 中列出的功能都已实现
- [ ] 每个页面都能正常访问和操作

## 技术验收
- [ ] `docker compose up` 一键启动，前后端都正常
- [ ] 前端 `npm run build` 零错误零警告
- [ ] 后端所有接口在 Swagger /docs 中可正常测试
- [ ] 数据库所有表都有 RLS 策略
- [ ] 没有 console.log / print 残留
- [ ] .env 文件没有提交到 Git（在 .gitignore 中）

## QA 测试验收（按 qa-testing.md 执行）
- [ ] 冒烟测试全部通过（首页加载、API 健康、控制台干净）
- [ ] 核心业务流程 E2E 测试通过（Browser MCP 截图为证）
- [ ] 响应式测试通过（手机/平板/桌面三个视窗）
- [ ] 健康评分 ≥ 70 分
- [ ] 所有 Bug 已修复或标记为延迟修复（附原因）

## UI 验收（按 ui-design.md 设计 QA 清单逐项检查）
- [ ] Light / Dark 模式都正常
- [ ] 手机端（375px 宽）布局正常
- [ ] 毛玻璃效果、动效都在运行
- [ ] 所有按钮、输入框有 hover/focus 反馈
- [ ] 色彩体系正确（Rose/Sky/Amber，无紫色）
- [ ] 间距遵循 8px 网格，圆角体系一致
- [ ] 无障碍：对比度 ≥ 4.5:1、装饰元素 aria-hidden

## 安全验收
- [ ] 前端代码中无 SERVICE_ROLE_KEY
- [ ] 所有写操作都需要登录
- [ ] 用户只能操作自己的数据（RLS + 后端双重校验）
- [ ] 无 SQL 注入风险（全部参数化 / 使用 Supabase SDK）
```

## 八、常用对话模板（直接复制粘贴给 AI）

### 启动新项目
```
我要开发一个新项目。
请先阅读 grules/ 目录下的所有文件，了解我的技术栈和规范。
然后阅读 product/ 目录下的需求文档。
按照 task-workflow.md 的流程，先帮我做需求分析（不要写代码）。
```

### 确认方案后拆任务
```
方案我确认了。请帮我拆解成开发任务列表。
按照 task-workflow.md 中的格式输出。
```

### 开始某个任务
```
开始任务 T-X.X。
请严格遵守 grules/ 下的所有编码规范。
完成后输出交付清单。
```

### 任务出了问题
```
任务 T-X.X 有问题：[描述问题]。
请检查并修复，修复后重新输出交付清单。
```

### 全部完成，做最终检查
```
所有任务已完成。
请按照 task-workflow.md 中的集成验收检查清单逐项检查。
报告每一项的通过/不通过状态。
```
