> 后端：自托管 Supabase Auth（GoTrue）；前端：`@supabase/supabase-js` v2。

## 一、登录

### 1.1 邮箱 + 密码

```
1. 用户在 /auth/login 提交 { email, password }
2. 前端调用 supabase.auth.signInWithPassword({ email, password })
   → 直连 Supabase Auth /token?grant_type=password
3. 成功：返回 { access_token, refresh_token, user }
   - supabase-js 自动写入 localStorage['zhiyu-auth']
   - 前端 onAuthStateChange 监听，写 Zustand authStore（id / email / role / avatar）
   - 同步调用我方 Hono API POST /api/auth/session-register，记录 user_sessions 行（详见 §四 多设备）
   - 检查 super_admin → 跳 /admin；user → 跳 redirect 参数或 /
4. 失败：
   - invalid_credentials → AUTH_INVALID_CREDENTIALS（"邮箱或密码错误"）
   - email_not_confirmed → AUTH_EMAIL_NOT_VERIFIED（"请先验证邮箱"）
   - 账号被禁用 → AUTH_ACCOUNT_DISABLED（"账号已停用，请联系客服"，见 §六）
   - 其他网络错 → AUTH_UNKNOWN
5. 凭证存储：localStorage（supabase-js 默认；本期沿用，见 99 Q4）
6. 后续请求：所有 /api/* 自动附带 Authorization: Bearer <access_token>（fetch 拦截器统一注入）
```

### 1.2 Google OAuth

```
1. 用户在 /auth/login 点 "Google 继续"
2. 前端调用 supabase.auth.signInWithOAuth({
     provider: 'google',
     options: { redirectTo: window.location.origin + '/auth/callback' }
   })
   → 浏览器跳转 Supabase Auth /authorize → Google
3. Google 同意后回跳 Supabase Auth → 再回跳 /auth/callback?code=...
4. /auth/callback 调用 supabase.auth.exchangeCodeForSession(code)
   - 拿到 session 后流程同邮箱登录的 step 3
   - 若是首次登录：Supabase Auth 自动创建 auth.users 行（见 05 §二）
   - 我方 DB Trigger 自动 insert profiles 行（见 04 §二）
5. 失败：
   - oauth_provider_error → AUTH_OAUTH_FAILED（带 provider 错误描述）
   - 用户拒绝 → 静默回 /auth/login
```

### 1.3 管理端登录限制

- `/admin/auth/login` 页面**仅展示邮箱密码输入框**，不展示 Google 按钮。
- 登录成功后立即检查 `app_metadata.role`；不为 `super_admin` → 立即 `signOut` + 报错 `AUTH_NOT_ADMIN`（“非管理员账号”）。
- **本期不上二步验证**（PM 决策变更：不做 TOTP）；超管账号依靠「强密码 + 登录节流 + 邮箱验证」保护，二步验证列入后续安全升级计划。

## 二、Token 策略

| 项目 | 方案 |
|------|------|
| Token 类型 | JWT（HS256，由 Supabase Auth 签发，密钥 = `JWT_SECRET` 环境变量）|
| Access Token 有效期 | **1 小时**（Supabase 默认；不修改）|
| Refresh Token 有效期 | **30 天**（滚动续期；每次 refresh 后旧 refresh_token 立即失效）|
| 存储位置 | **httpOnly Cookie**（PM 决策）：`zhiyu-at`（access）+ `zhiyu-rt`（refresh），都为 `HttpOnly; Secure; SameSite=Lax; Path=/`，顶级域名下跨子域共享 |
| Cookie 实现 | supabase-js 传入自定义 `auth.storage` adapter，读写调用同域代理接口 `POST /api/auth/cookie` 由 Hono 负责 Set-Cookie（详见 [03-权限校验机制.md §二.4](./03-权限校验机制.md)） |
| 刷新机制 | supabase-js 自动：access_token 到期前 60s 触发 refresh；失败 3 次 → 触发全局登出 |
| 携带方式 | 同域请求 → 浏览器自动携 Cookie；Hono 中间件从 `c.req.cookie('zhiyu-at')` 读取并作为 `Authorization: Bearer` 等价校验 |
| 后端验签 | Hono 中间件用 `jose` 校验签名 + `exp`，密钥从 `process.env.SUPABASE_JWT_SECRET` 读取 |
| CSRF 防护 | 启用 **Double-Submit Token**：登录后同时下发一个非 HttpOnly 的 `zhiyu-csrf` Cookie，前端每次写请求在 `X-CSRF-Token` Header 中回传；后端中间件 `csrfRequired` 校验两者一致 |
| Token Payload 关键字段 | `sub`（user_id, UUID）/ `email` / `app_metadata.role` / `exp` / `iat` |

> **不做** Token 黑名单。“踢人”通过 Supabase Admin API 撚销 refresh_token 实现（见 §四）；access_token 在自然过期前仍有效（最长 1h），可接受风险。

## 三、登出

```
1. 用户点击右上角“退出登录”
2. 前端调用 supabase.auth.signOut({ scope: 'local' })
   - 同域代理接口返回 Set-Cookie：zhiyu-at / zhiyu-rt / zhiyu-csrf 全部 Max-Age=0
   - Zustand authStore reset
3. 同时调用我方 POST /api/auth/session-revoke { sessionId } 删除 user_sessions 行
4. 跳转 /auth/login（应用端）或 /admin/auth/login（管理端）
5. 若调用 signOut 失败：忽略，依然清本地 + 跳页（已登出，再调一次也无副作用）
```

## 四、会话管理（多设备 3 上限）

### 4.1 策略

- 不自动登出（PM 决策）；只要 refresh_token 未过期（30d）就保持登录。
- 同一 `user_id` 最多 **3 个活跃会话**；第 4 次登录踢掉**最早创建**的会话。

### 4.2 实现：`user_sessions` 表 + 登录时扫描

详细表结构见 [04-数据结构.md §三](./04-数据结构.md)。

登录成功后流程：

```
POST /api/auth/session-register
  body: { device: 'web' | 'mobile-web', ua: navigator.userAgent }
  header: Authorization: Bearer <new_access_token>

后端处理（Hono handler）：
  1. authMiddleware 解析出 user_id
  2. 取 supabase.auth.admin.getUserSessions(user_id)（自建查询：select * from auth.refresh_tokens where user_id = ?）
     ※ 实际通过 service_role 客户端查 auth.refresh_tokens，因为 GoTrue 未暴露此 API
  3. 在我方 user_sessions 插入新行 { user_id, refresh_token_hash, device, ua, ip, created_at, last_seen_at }
  4. count > 3 → 选 created_at 最早的那条 oldest
       - 调 supabase.auth.admin.signOut(oldest.refresh_token) 撤销 refresh
       - 删除 user_sessions 中的 oldest 行
  5. 返回 { sessionId }
```

### 4.3 被踢端的体验

- 被踢端的 access_token 仍然有效（最长 1h），但下次刷新会失败 → supabase-js 触发 `onAuthStateChange('SIGNED_OUT')` → 全局登出 → 跳 `/auth/login?reason=kicked`，登录页 Toast：**"您的账号在其他设备登录，已被退出"**。
- 应用端可选优化：authStore 启动时调 `GET /api/auth/session-status`，若返回 `{ valid: false, reason: 'kicked' }` 立即登出。

### 4.4 会话记录仅供后台踢人使用

- `user_sessions` 表仅供（1）登录时检查会话数 ≥ 3 踢最早；（2）管理端「禁用账号」时撚销该用户全部会话（§六）。
- **不对用户提供「我的设备」页面**（PM 决策：有问题重新登录即可，减少 UI 复杂度）；也不上「心跳接口 + last_seen」。`last_seen_at` 字段仅在用户主动调需鉴权接口时顺手更新。

## 五、密码安全

| 项目 | 方案 |
|------|------|
| 存储方式 | bcrypt（Supabase Auth 默认 cost=10）；我方代码**不接触明文密码** |
| 密码规则 | 最少 8 位；必须包含字母 + 数字；前端 zod 校验 + Supabase Auth 服务端再校一次 |
| 注册时强度提示 | 弱 / 中 / 强（基于长度 + 字符多样性，纯前端展示，不阻塞）|
| 登录失败限制 | **连续 5 次失败锁定 15 分钟**（按 `email + ip` 维度，记录在 `auth_login_attempts` 表，见 04 §四）|
| 锁定解除 | 自动到期；管理端可手动重置（删除 `auth_login_attempts` 行）|
| 防爆破补充 | **本期不接第三方验证码（PM 决策）**；仅靠本站后端节流（锁定 + 邮箱重设 60s/1h 节流 + 注册 IP 1h ≤ 5 次）|

> Hono 中间件 `loginRateLimit` 在 `POST /api/auth/login-attempt-record` 上前置；Supabase Auth 本身不做按 email 限流，需我方在 Hono 层包一层（详见 §六.2）。

## 六、被禁用账号

### 6.1 标记

- `profiles.is_disabled boolean default false`（详见 04 §二）。
- 管理端在用户管理页可「禁用 / 启用」，并填禁用原因 `disabled_reason`。
- **禁用动作的联合效果（PM 决策）**：
  1. `update profiles set is_disabled=true, disabled_reason=?, disabled_by=?, disabled_at=now() where id=?`；
  2. 调 `supabase.auth.admin.signOut(userId, { scope: 'global' })` 撚销该用户全部 refresh_token；
  3. `delete from user_sessions where user_id=?`；
  4. 清 `disabledCache.delete(userId)` 让其他 Hono 节点迅速感知；
  5. 写 `audit_logs` 记录 `user.disable`。
- **启用动作**：`update profiles set is_disabled=false, disabled_reason=null where id=?` + 清缓存 + 写 `user.enable`；用户下次重新登录即恢复正常。

### 6.2 拦截位置

- **登录前置**：`POST /api/auth/login-attempt-record`（Hono 路由，前端在调 supabase.auth.signInWithPassword 之前先调）
  - 查 `profiles where email = ?` → 若 `is_disabled = true`，返回 `AUTH_ACCOUNT_DISABLED`，前端不再调 supabase。
- **Token 校验**：authMiddleware 在每次请求时查一次 `profiles.is_disabled`（带 30s 内存缓存以减少 DB 压力，详见 03 §二）；若 `true`，返回 `AUTH_ACCOUNT_DISABLED` + 强制登出。

### 6.3 前端展示

- 登录页 Toast：**"账号已被停用，原因：{disabled_reason}。如有疑问请通过客服工单联系我们。"**（含跳转到 `/help/contact` 的链接）
- 已登录用户被禁用 → 下一次 API 调用收到 `AUTH_ACCOUNT_DISABLED` → 全局登出 → 跳同上提示页。

## 七、忘记密码

### 7.1 流程

```
1. 用户在 /auth/forgot 输入 email，提交
2. 前端调 supabase.auth.resetPasswordForEmail(email, {
     redirectTo: 'https://app.zhiyu.com/auth/reset-password'
   })
   → Supabase Auth 发送邮件（模板自托管邮件 SMTP，详见 G1 06 §部署）
3. 即使 email 不存在也返回成功（防枚举），前端统一显示"如该邮箱已注册，重置链接已发送"
4. 用户点击邮件链接 → 跳 /auth/reset-password?access_token=...&type=recovery
5. /auth/reset-password 页：
   - 自动调 supabase.auth.exchangeCodeForSession 进入 recovery session
   - 用户输入新密码 + 再次确认
   - 调 supabase.auth.updateUser({ password: newPwd })
   - 成功 → 自动登入并跳 /me；**同时撚销该 user 的所有其他 refresh_token**（PM 决策：所有其他设备需重新登录），并同步 `delete from user_sessions where user_id=? and id<>currentSessionId`
6. 失败：
   - 链接过期 / 已使用 → AUTH_RESET_TOKEN_INVALID（“链接已失效，请重新申请”）
```

### 7.2 频次限制

- 同一 email 60 秒内最多发 1 次；同一 IP 1 小时最多发 5 次。
- 实现：Hono 路由 `POST /api/auth/forgot-password-throttle`，前端调 supabase 之前先调本接口；throttle 状态用 Redis（key=`forgot:{email}` / `forgot:ip:{ip}`，TTL 60s / 3600s）。

## 八、邮箱验证

- Supabase Auth 默认要求邮箱验证才能登录（`MAILER_AUTOCONFIRM=false`，PM 决策）。
- Google 注册的账号视为已验证（Supabase 自动 set `email_confirmed_at`）。
- **本期邮件发送 = Mock**（与 G1 「缺少外部 SaaS 均使用 mock adapter」策略一致）：dev / Docker 环境下邮件内容写入 `system/.dev/mailbox/{email}-{timestamp}.eml` 且控制台打印验证链接；prod 接入真实 SMTP 后端。
- 邮箱注册流程详见 [05-注册流程.md §一](./05-注册流程.md)。
