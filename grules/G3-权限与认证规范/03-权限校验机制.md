## 一、前端

### 1.1 路由守卫（TanStack Router）

- 在 `system/apps/web-app/src/routes/_authed.tsx` 与 `system/apps/web-admin/src/routes/_admin.tsx` 各定义一个**布局路由**作为守卫：

  ```ts
  // _authed.tsx（应用端：要求登录）
  export const Route = createFileRoute('/_authed')({
    beforeLoad: ({ context, location }) => {
      const { user } = context.auth;
      if (!user) {
        throw redirect({ to: '/auth/login', search: { redirect: location.href } });
      }
    },
    component: Outlet,
  });

  // _admin.tsx（管理端：要求 super_admin）
  export const Route = createFileRoute('/_admin')({
    beforeLoad: ({ context }) => {
      const { user } = context.auth;
      if (!user) throw redirect({ to: '/admin/auth/login' });
      if (user.role !== 'super_admin') throw redirect({ to: '/admin/auth/login', search: { reason: 'not_admin' } });
    },
    component: AdminLayout,
  });
  ```

- 受保护路由放在 `_authed/` / `_admin/` 子目录下，文件路由自动继承。
- 公开路由（`/auth/*`、`/discover` 前 3 主题、404）放在守卫外。

### 1.2 菜单过滤

- 菜单源数据 `MENU_ITEMS`（在 `packages/shared/src/menus.ts`）每项带 `requireAuth?: boolean` / `requireRole?: 'super_admin'`。
- `<TopNav />` 渲染时按当前 `user.role` 过滤；未登录看应用端公共菜单（参考 G2 03 §1.2）。
- **不做**按钮级 RBAC 配置；按钮可见与否在组件内部按 `useAuth()` 简单判断。

### 1.3 按钮 / 操作级控制

- 原则：**无权限按钮直接不渲染（隐藏）**；不做"灰显禁用"。
- 例外：用户已登录但**所有权不匹配**（如别人帖子的"删除"按钮）→ 不渲染。
- 危险操作（删除 / 封禁）即便按钮可见，提交时后端二次校验，前端不可信。

### 1.4 Token 过期处理

```
supabase-js auto refresh 失败
  → onAuthStateChange('SIGNED_OUT')
  → authStore.reset()
  → 全局 fetch 拦截器收到 401 / AUTH_TOKEN_EXPIRED
  → 弹 Toast "登录已过期，请重新登录"
  → router.navigate('/auth/login', { search: { redirect: currentPath } })
```

- TanStack Query 全局 `queryClient.setDefaultOptions({ queries: { retry: (n, err) => err?.code !== 'AUTH_TOKEN_EXPIRED' && n < 2 } })` 避免无限重试。

### 1.5 全局 `useAuth` 契约

```ts
type AuthState = {
  user: { id: string; email: string; role: 'user' | 'super_admin'; avatar?: string } | null;
  loading: boolean;
  signIn: (email: string, password: string) => Promise<void>;
  signInWithGoogle: () => Promise<void>;
  signOut: () => Promise<void>;
};
```

由 Zustand 实现，挂在 `packages/shared/src/auth/useAuth.ts`，应用端 / 管理端共用。

## 二、后端

### 2.1 中间件分层

| 中间件 | 作用 | 失败时返回 |
|--------|------|-----------|
| `authRequired` | 从 **`zhiyu-at` Cookie**（优先）或 `Authorization: Bearer` Header 读取 JWT；校验签名 · `exp` · `profiles.is_disabled`；写 `c.set('user', {...})` | 401 `AUTH_TOKEN_MISSING` / `AUTH_TOKEN_EXPIRED` / `AUTH_ACCOUNT_DISABLED` |
| `adminRequired` | 在 `authRequired` 之后；要求 `user.role === 'super_admin'` | 403 `AUTH_FORBIDDEN` |
| `csrfRequired` | POST / PUT / PATCH / DELETE 路由必加；比对 `X-CSRF-Token` Header 与 `zhiyu-csrf` Cookie（Double-Submit） | 403 `AUTH_CSRF_INVALID` |
| `optionalAuth` | 解析 JWT 但不强制；用于“未登录可浏览”的路由 | 不报错；`c.get('user')` 可能为 `null` |

注册示例：

```ts
const app = new Hono();
app.use('/api/admin/*', authRequired, adminRequired);
app.use('/api/me/*', authRequired);
app.use('/api/discover/*', optionalAuth);
```

### 2.2 `authRequired` 实现要点

```ts
import { jwtVerify, createRemoteJWKSet } from 'jose';

const SUPABASE_JWT_SECRET = process.env.SUPABASE_JWT_SECRET!;
const KEY = new TextEncoder().encode(SUPABASE_JWT_SECRET);
const disabledCache = new LRU<string, boolean>({ max: 5_000, ttl: 30_000 }); // 30s

export const authRequired: MiddlewareHandler = async (c, next) => {
  // 1. 优先从 HttpOnly Cookie 读取；后兼容 Authorization Header（服务间调用场景）
  const cookieToken = getCookie(c, 'zhiyu-at');
  const authHeader = c.req.header('authorization');
  const token = cookieToken ?? (authHeader?.startsWith('Bearer ') ? authHeader.slice(7) : null);
  if (!token) return jsonError(c, 401, 'AUTH_TOKEN_MISSING');
  let payload;
  try {
    ({ payload } = await jwtVerify(token, KEY));
  } catch (e) {
    const code = e?.code === 'ERR_JWT_EXPIRED' ? 'AUTH_TOKEN_EXPIRED' : 'AUTH_TOKEN_INVALID';
    return jsonError(c, 401, code);
  }
  const userId = payload.sub as string;
  let disabled = disabledCache.get(userId);
  if (disabled === undefined) {
    const { data } = await supabaseAdmin.from('profiles').select('is_disabled').eq('id', userId).single();
    disabled = !!data?.is_disabled;
    disabledCache.set(userId, disabled);
  }
  if (disabled) return jsonError(c, 401, 'AUTH_ACCOUNT_DISABLED');
  c.set('user', {
    id: userId,
    email: payload.email,
    role: (payload.app_metadata as any)?.role ?? 'user',
  });
  await next();
};
```

- 缓存目的：避免每个请求都查一次 DB；30s 内被禁用的用户最多继续访问 30s，业务可接受。
- 主动清缓存：管理端禁用账号后调 `disabledCache.delete(userId)`（通过内存 Bus / Redis pubsub 跨 Hono 进程同步，本期单实例先用本地）。

### 2.3 RLS（行级安全）兜底

- Supabase 表默认开 RLS；`profiles` / `user_sessions` / 业务表均按 `auth.uid()` 限制读写。
- Hono 后端使用 `service_role` key（绕 RLS）；**仅在通过 `authRequired` 校验后**用 `supabaseAdmin` 操作；不直接转发用户 `access_token` 给 Supabase（避免双重校验开销）。
- 公网直连 Supabase（前端 anon key）：仅允许通过 RLS 安全暴露的视图 / 表（本期暂不开放，所有业务都走 Hono）。

### 2.4 Cookie 代理接口（supabase-js 适配器配套）

- 前端 supabase-js 初始化时传入 `auth.storage` 自定义实现，**不读写 localStorage**；读写都发送到同域 Hono：
  - `GET /api/auth/cookie/get` → 仅返回 `{ accessToken: present(boolean), refreshToken: present(boolean) }`（不回传实际 token，不破坏 HttpOnly）+ user payload（后端解 JWT 后仅取 `id/email/role`）。
  - `POST /api/auth/cookie/set` body: `{ access_token, refresh_token }` → Set-Cookie：`zhiyu-at`（HttpOnly，Max-Age=3600） + `zhiyu-rt`（HttpOnly，Max-Age=2592000） + `zhiyu-csrf`（非 HttpOnly，随机 32 字节）。
  - `POST /api/auth/cookie/clear` → 三个 Cookie 全部 Max-Age=0。
- supabase-js 调 `signInWithPassword` / `exchangeCodeForSession` / `refreshSession` 后，在适配器中拦截并调用 `cookie/set` 同步到后端。

## 三、统一错误格式

所有 4xx / 5xx 响应使用相同 schema（与 G1 04-API §五 错误响应保持一致）：

```json
{
  "error": {
    "code": "AUTH_TOKEN_EXPIRED",
    "message": "登录已过期，请重新登录",
    "details": null
  }
}
```

- `message` 由后端生成，i18n 由前端按 `code` 重新映射（不直接展示后端 message，除非 `code === 'AUTH_OAUTH_FAILED'` 等需要透传 provider 信息的场景）。
- HTTP 状态码：`401` = 未认证 / 登录过期 / 账号禁用；`403` = 已登录但权限不足。

## 四、AUTH 错误码清单

| Code | HTTP | 触发场景 | 前端默认提示（zh）|
|------|------|---------|------------------|
| `AUTH_TOKEN_MISSING` | 401 | 请求无 Authorization header | 请先登录 |
| `AUTH_TOKEN_INVALID` | 401 | JWT 签名错 / 格式错 | 登录已失效，请重新登录 |
| `AUTH_TOKEN_EXPIRED` | 401 | JWT exp 过期 | 登录已过期，请重新登录 |
| `AUTH_INVALID_CREDENTIALS` | 401 | 邮箱 / 密码错 | 邮箱或密码错误 |
| `AUTH_EMAIL_NOT_VERIFIED` | 401 | 注册后未点验证邮件 | 请先验证邮箱 |
| `AUTH_ACCOUNT_DISABLED` | 401 | profiles.is_disabled = true | 账号已被停用，原因：{reason}。请联系客服 |
| `AUTH_NOT_ADMIN` | 403 | 非 super_admin 进管理端 | 该账号无管理员权限 |
| `AUTH_FORBIDDEN` | 403 | 角色不足 / 资源所有权不符 | 没有访问权限 |
| `AUTH_OAUTH_FAILED` | 400 | Google OAuth 异常 | Google 登录失败：{provider_error} |
| `AUTH_RESET_TOKEN_INVALID` | 400 | 重置链接过期 / 已用 | 链接已失效，请重新申请 |
| `AUTH_LOGIN_RATE_LIMITED` | 429 | 5 次失败 / 15 分钟内 | 失败次数过多，请 {minutes} 分钟后重试 |
| `AUTH_FORGOT_RATE_LIMITED` | 429 | 60s / 1h throttle 触发 | 操作过于频繁，请稍后再试 |
| `AUTH_SESSION_KICKED` | 401 | 被其他设备挤掉 | 您的账号在其他设备登录，已被退出 |
| `AUTH_SUPER_ADMIN_SELF_DELETE` | 400 | 超管尝试删除自己 | 超级管理员账号不能删除自己 |
| `AUTH_EMAIL_TAKEN` | 400 | 注册时邮箱已存在 | 该邮箱已注册，请直接登录或找回密码 |
| `AUTH_WEAK_PASSWORD` | 400 | 密码不符合规则 | 密码不符合规则，请重新输入 |
| `AUTH_CSRF_INVALID` | 403 | CSRF Header 与 Cookie 不一致 | 页面已过期，请刷新重试 |

## 五、并发与一致性

- `authRequired` 缓存被禁用状态 30s；管理端禁用动作记录到 `audit_logs`，方便排查。
- 多实例 Hono 部署时（未来）切换 disabled cache 为 Redis（key=`auth:disabled:{userId}`，TTL 30s）。
- Token 撤销同样依赖 Supabase Auth 自带机制；我方不维护单独黑名单。
